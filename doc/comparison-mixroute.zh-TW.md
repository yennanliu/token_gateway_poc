# 本專案 vs. MixRoute — 差異說明（繁體中文）

> 內部分析文件
> 參考對象：<https://mixroute.ai/zh-hant/>
> 撰寫日期：2026-07-07
> 英文版與「可借鏡的點子清單」請見 [`comparison-mixroute.md`](./comparison-mixroute.md)

本文說明本專案（`token_gateway_poc`，一套**自架的 LLM API 閘道**）與 **MixRoute**（一項**託管型 SaaS 聚合閘道**）之間的核心差異，協助判斷兩者的定位與適用情境。

---

## 一句話總結

- **MixRoute** 是「**一把金鑰、串接所有廠商**」的**託管服務**，主打**模型廣度**、**毫秒級自動容錯**與**零加價**的用量後付制。它是要**直接使用的產品**。
- **本專案** 是一套「**預付點數 + 多租戶控管**」的**可自架閘道**，主打**精準計費帳本**、**組織/租戶權限控管**與**完全可掌控、可自架、有測試**。它是要**自己營運的基礎建設**。

兩者不是同一類東西：一個是「你去用的雲端服務」，一個是「你自己架起來賣或自用的系統」。

---

## MixRoute 是什麼

一項託管式的統一 AI API 聚合閘道：

- **模型廣度**：200+ 模型、20+ 廠商（OpenAI、Anthropic、Google、Mistral、Meta、Cohere、DeepSeek、Perplexity、xAI、Bedrock、Azure…）。
- **API 相容性**：**相容 OpenAI SDK**，只需把 `base_url` 指到 MixRoute，即可用同一組憑證切換不同模型。
- **路由與可靠性**：跨模型動態路由（依速度／成本／品質）、廠商故障時**毫秒級自動容錯**、**預留產能（provisioned throughput）**跳過公有佇列、跨時區全球排程（亞洲／歐洲／美洲三區）。
- **計費模式**：**用量後付制**，按每百萬 token 計價，**官方定價、零加價**，統一帳單、無需綁卡，30 秒開通。
- **隱私**：僅收集中繼資料（token 數、延遲、成本），**提示內容一律不落地、不記錄、不用於訓練**，純記憶體處理。
- **對象**：需要正式環境等級 AI 基礎建設、又不想自行管理多個廠商帳號與尖峰限流問題的企業與開發團隊。

## 本專案是什麼

一套 test-first 打造的自架 LLM 閘道（`src/gateway/`，約 2,527 行程式、91 個通過測試、CI 具遷移漂移防護、Docker + Alembic）：

- **多廠商代理**：OpenAI／Anthropic／Gemini，**各自提供原生格式端點**，並具 OpenAI→Anthropic 的格式轉譯。
- **預付點數帳本**：整數 micros 精度（1,000,000 micros = 1 credit = $0.01），**原子扣款**（UsageEvent + 負向 LedgerEntry + 工作區餘額扣減，單一交易完成），餘額不可為負。
- **多租戶階層**：組織 → 工作區 → 專案 → 金鑰（Org → Workspace → Project → Key），並具 owner／admin／member 三級 RBAC、使用者 session、稽核日誌。
- **限流**：每金鑰 RPM，記憶體 token bucket 或 Redis 固定視窗。
- **月度預算**：每工作區當月累計用量上限（硬性擋下）。
- **金流**：Stripe Checkout（含 mock 模式與簽章 webhook）儲值。
- **監控**：Prometheus `/metrics`、請求日誌、用量事件、Vue 3 主控台。
- **部署**：可完全自架（Docker/compose、Postgres/Redis、Alembic 遷移）。

## 三個關鍵差異（重點在這）

### 1. 計費哲學：預付硬扣 vs. 後付結算

這是最本質的差異。

- **本專案**：**預付**。先儲值點數，扣款在請求**前置檢查**、且與用量記錄在**同一筆交易原子完成**，餘額不可為負。金額全程用**整數 micros**，可精準對帳、不會有浮點誤差。這是「代幣／點數閘道」的核心價值。
- **MixRoute**：**後付**。用量事後累計、統一出帳、零加價。使用者體驗簡單，但它是「用完再付」，不是「先儲值、扣到零就擋」的硬性額度控管。

若你的產品定位是**預付、精準、硬性額度**的代幣閘道，本專案的模型才對；MixRoute 的後付模型是另一種產品思路。

### 2. 廣度與可靠性基礎建設：MixRoute 明顯領先

MixRoute 的 200+ 模型、毫秒級容錯、預留產能、跨時區排程，是**投入大量營運基礎建設**才能達成的。本專案目前只有同端點重試，沒有跨廠商備援、金鑰輪替或負載平衡。若需求是「一把金鑰打遍天下、且高可用」，MixRoute 現成即用；本專案要達到同等廣度需要投入開發（見 `comparison-mixroute.md` 的點子清單：模型註冊表、多金鑰輪替、跨廠商備援鏈）。

### 3. 控制平面與可掌控性：本專案明顯領先

本專案有**完整的多租戶控制平面**（組織／工作區／專案／金鑰四層階層、三級 RBAC、月度預算、稽核日誌、Stripe 儲值），而且**完全可自架、開源可改、有 91 個測試護欄**。MixRoute 的多租戶與權限屬企業方案，細節不透明，且資料與計費的「真相來源」在對方手上。若你需要**自己擁有租戶、金鑰、金流與帳本**（例如要對下游客戶再賣、或有資料自主／合規要求），本專案這條路才走得通。

## 何時選哪一個

**選 MixRoute（直接用託管服務）當：**
- 你只是要**盡快用上很多模型**、且要高可用容錯，不想自己維運。
- 你可接受**後付、統一帳單**的計費方式。
- 你不需要自己掌控租戶／金鑰／帳本，也沒有強烈的資料自主或合規需求。

**選本專案（自架這套閘道）當：**
- 你要做**預付點數／代幣制**、需要**精準到分的硬性額度**與可對帳帳本。
- 你要對**多租戶（組織／工作區／專案）與 RBAC** 有完整掌控，甚至要**再賣給下游**。
- 你需要**自架、可改、資料自主、可自訂費率與加價**。
- 你能接受目前僅 3 廠商、需要時再自行擴充廣度與容錯（有明確的擴充路徑）。

> **一句話決策**：要「**現成的模型廣度與高可用**」→ MixRoute；要「**自己擁有金流、租戶與帳本的可自架閘道**」→ 本專案。兩者甚至可並存——本專案負責租戶／點數／控管，底層轉發未來也可接上像 MixRoute（或 LiteLLM）這類廣度來源（詳見 [`litellm-evaluation.md`](./litellm-evaluation.md)）。

## 可借鏡 MixRoute 的方向（摘要）

完整、對應到程式碼的點子清單見英文版 [`comparison-mixroute.md`](./comparison-mixroute.md)。優先順序摘要：

1. **模型 → 廠商 動態路由註冊表**（把 3 廠商變成 N 廠商的關鍵解鎖）。
2. **每廠商多金鑰輪替**（最便宜的可靠性提升）。
3. **明文承諾並用測試保證「零落地」隱私**（我們本來就沒存內容，把它變成明確保證）。
4. **可設定的加價／利潤率**（費率表已存在，讓營運者可選零加價或加成）。
5. **跨廠商備援鏈**（把重試延伸為跨廠商切換）。
6. **模型別名／虛擬模型**（`auto`／`cheapest`／`fastest`）。
7. **更豐富的分析**（p50/p95 延遲、成本走勢——`latency_ms` 已在收集，只差呈現）。
